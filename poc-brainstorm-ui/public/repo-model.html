Where does the code actually live?

Your product uses a graph for navigation and structure. The code still lives in normal files on disk — like any Git repo. The graph points at those files; it does not replace them.

One sentence: Disk = the real source text. Graph = an index (metadata) that says which files belong to which “node” and how capsules connect.

Shared code (the confusing part)

The orange file exists once. Both Account and Settings import it. In the graph, you might show several leaves that all “point at” that same file (or one definition leaf + use-sites). That’s the “linked leaf” idea — not three copies of the source on disk.

What is graph.yaml?

A small file (name is an example) that lists nodes, parent/child, optional conceptual edges, and anchor paths — the strings that tie each leaf to real paths under src/. It is not where you paste your React components.

# conceptual example — not exact syntax nodes: ui: { kind: inner, children: [account-form, settings-panel, shared-field] } account-form: { kind: leaf, anchor: "src/ui/features/account/AccountForm.tsx" } settings-panel:{ kind: leaf, anchor: "src/ui/features/settings/SettingsPanel.tsx", sharedGroup: "text-1" } shared-field: { kind: leaf, anchor: "src/ui/components/SharedTextField.tsx", sharedGroup: "text-1" }

Remember

  • npm test, TypeScript, ESLint see normal files — they don’t need the graph.
  • The graph is for your product UI, agents, and “where does this feature live?”
  • Shared code = one file, many importers; the graph shows the relationship so you don’t think they’re unrelated.
